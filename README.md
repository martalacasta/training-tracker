# Training Tracker

Personal training dashboard with:
- Strava ingestion pipeline (webhook-first + scheduled reconciliation).
- Goal-aware coach memory.
- Recommendation and insight generation.
- Static UI deployment on GitHub Pages.

## Architecture

1. Strava webhook receives activity `create/update/delete` events.
2. Sync script reconciles new/changed activities and writes canonical JSON to `public/data/`.
3. Coach memory script computes adherence, phase, and fatigue flags.
4. Recommendation script updates next sessions (rule-based or optional LLM) and writes adaptation trace metadata.
5. Insights script writes summary text used by the dashboard.
6. Optional Worker `/chat` endpoint grounds coach-chat answers on generated data files.
7. GitHub Pages serves the static frontend.

## Local setup

```bash
npm install
cp .env.example .env
```

Set in `.env`:
- `STRAVA_CLIENT_ID`
- `STRAVA_CLIENT_SECRET`
- `STRAVA_REFRESH_TOKEN`

Optional Strava retry tuning:
- `STRAVA_RETRY_MAX` (defaults to `4`)
- `STRAVA_RETRY_BASE_MS` (defaults to `1500`)

Optional LLM:
- `LLM_API_URL`
- `LLM_API_KEY`
- `LLM_MODEL`

Optional GitHub Models:
- `GITHUB_MODELS_MODEL`
- `GITHUB_MODELS_API_KEY` (or `GITHUB_TOKEN` in CI)
- `GITHUB_MODELS_API_URL` (optional override, default: `https://models.github.ai/inference/chat/completions`)

Optional frontend chat:
- `VITE_CHAT_API_URL` (full URL of worker `/chat` endpoint)

For GitHub Actions repository **variables**, names cannot start with `GITHUB_`.
Use `MODELS_MODEL` and optional `MODELS_API_URL` in Actions, which are mapped by workflow to runtime env vars.

Optional per-activity advice scope:
- `ACTIVITY_ADVICE_LOOKBACK_DAYS` (defaults to `7`)
- `ACTIVITY_ADVICE_REGENERATE_STALE` (defaults to `false`)
- `ACTIVITY_ADVICE_MAX_PER_RUN` (defaults to `20`)
- `ACTIVITY_ADVICE_CONTEXT_LIMIT` (defaults to `8`)

## Cloudflare Worker setup (chat + webhook)

Use this when you want the `/chat` endpoint (and optional webhook endpoint) deployed server-side.

```bash
# 1) Install Wrangler once
npm i -g wrangler

# 2) Authenticate
wrangler login

# 3) From repo root, run local worker dev
wrangler dev webhook/worker.ts
```

Set worker secrets:

```bash
# LLM provider (recommended)
wrangler secret put LLM_API_URL
wrangler secret put LLM_API_KEY
wrangler secret put LLM_MODEL

# Optional GitHub Models alternative
wrangler secret put GITHUB_MODELS_API_KEY
```

Set worker vars (via dashboard or wrangler config):
- `GITHUB_MODELS_MODEL` (if using GitHub Models)
- `GITHUB_MODELS_API_URL` (optional override)
- `CHAT_DATA_BASE_URL` (e.g. `https://<your-pages-domain>/data`)
- `CHAT_ALLOWED_ORIGIN` (e.g. `https://<your-pages-domain>`)
- `STRAVA_WEBHOOK_VERIFY_TOKEN` (if using webhook verification)

Deploy worker:

```bash
wrangler deploy webhook/worker.ts
```

Then set frontend env:

```bash
VITE_CHAT_API_URL=https://<your-worker-domain>/chat
```

## GitHub setup checklist

1. Create Strava API app and set callback domain for webhook service.
2. Deploy `webhook/worker.ts` (Cloudflare Worker, Vercel, or similar) and register Strava subscription.
3. Add repository secrets:
   - `STRAVA_CLIENT_ID`
   - `STRAVA_CLIENT_SECRET`
   - `STRAVA_REFRESH_TOKEN`
   - `LLM_API_URL` / `LLM_API_KEY` / `LLM_MODEL` (optional)
   - repository variable `MODELS_MODEL` (optional GitHub Models alternative, example value: `openai/gpt-4.1` or `openai/gpt-4o-mini`)
   - no extra GitHub Models secret is required in workflow when using `${{ github.token }}` with `permissions: models: read`
4. Enable GitHub Pages in repository settings.
5. Let `sync-reconcile.yml` update data daily and `deploy.yml` publish frontend.

## Notes

- `public/data/` is the contract between backend scripts and frontend.
- `public/data/next-recommendations.json` includes optional `trace` metadata (run id, source/model, week date range, run/gym weekly progress, and changes vs previous run) used by the UI.
- Coach chat uses a server-side Worker endpoint and never exposes LLM secrets in the browser.
- Per-activity coaching files are written to `public/data/activity-advice/<activityId>.json`.
- Athlete preferences/context can be set in `public/data/athlete-profile.json` and are included in advice prompts.
- If no LLM vars are set, recommendations are generated by deterministic rules.
- If no LLM/GitHub Models vars are set, activity advice generation is skipped safely.
- Activity advice generation is scoped by default to recent activities (7 days), no stale regeneration, capped per run.
- The pipeline is designed so recommendations carry context from historical activities and goals.
- The dashboard weekly section shows progress for fixed weekly targets: **4 run sessions** and **2 gym sessions** (gym = HIIT or WeightTraining from Strava sport types).
- Adaptation details are shown as a compact user summary, with technical trace available in a collapsible view.
